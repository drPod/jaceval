# Designing a v0 Jac skill eval: a defensible recipe

**Build a paired A/B harness of ~40–60 hand-written Jac tasks, run K≥5 samples per task per arm, grade correctness by executing `jac run` against hidden unit tests (40%) and grade idiomaticity with a rubric-based LLM judge that scores a binary checklist of Jac-native constructs before emitting a 1–5 overall score (60%), and report paired deltas with bootstrap CIs.** This is the highest-confidence setup given the research: it mirrors MultiPL-E's test harness (which is the only prior art that worked for truly low-resource languages), borrows Prometheus/G-Eval's rubric+CoT judge pattern, follows Miller-style paired inference so a small eval can still produce credible deltas, and uses Anthropic's progressive-disclosure Skills spec to define the context doc being tested. **Critically, there is no prior published Jac-codegen benchmark** — you are building v0 of a missing artifact, so you should not expect to find a harness to fork; you *can* and should fork MultiPL-E's translator/runner pattern and Inspect AI's Task/Scorer abstraction. Below are the specific design choices with citations, the pitfalls that will hurt you, and where research consensus is strong versus contested.

## What the research actually says

### LLM-as-judge for code: strong consensus on scaffolding, weak consensus on numbers
Zheng et al. ("Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena," NeurIPS 2023, arXiv:2306.05685) established the three patterns still in use: pairwise, single-answer grading, and reference-guided grading. GPT-4 hits ~80% agreement with humans — matching human-human agreement — but only when biases are mitigated. **Position bias is severe (10–40% verdict flips on swap) and the field-standard fix is to evaluate (A,B) and (B,A) and only count consistent wins as decisions, otherwise tie** (PandaLM, arXiv:2306.05087). Length/verbosity bias inflates scores for longer outputs by ~15%. Self-preference bias (~10% boost for same-family outputs) is real but Wataoka et al. (arXiv:2410.21819) argue it's really a perplexity-of-familiar-text effect — which is directly dangerous for Jac since Python-transliterated patterns will have lower perplexity than idiomatic walker code. **Mitigation: use a different model family as judge than generator, and explicitly instruct the judge to penalize Python-ish patterns.**

For code specifically, the best-validated reference-free judge is **CodeJudge** (Tong & Zhang, EMNLP 2024, arXiv:2410.02184, github.com/VichyTong/CodeJudge), which uses a two-stage "analyze-then-summarize" pipeline with a fault taxonomy and reports Kendall τ up to 0.502 — outperforming ICE-Score (Zhuo, EACL 2024, arXiv:2304.14317) by 12–42% correlation. But **recent work warns that LLM judges can catastrophically fail on code**: arXiv:2507.16587 reports Cohen's κ below 0.10 in several settings versus ground-truth execution, and arXiv:2508.12358 shows elaborate CoT prompts inflate false negatives ("fault-finding bias"). Consensus remedy: **keep correctness in execution-based territory and reserve the LLM judge for idiomaticity**, which is the one axis where no programmatic ground truth exists.

Binary or very short Likert scales beat 1–10. **Hamel Husain and Eugene Yan both insist on this**: nobody agrees what "3 vs 4" means, and 1–4 or binary cuts verbosity inflation and position inconsistency. When you need a richer scale, use the Prometheus pattern (Kim et al., ICLR 2024, arXiv:2310.08491) — explicit *descriptors per level*, reference answer anchored to score=5, structured output with feedback field then `[RESULT] X`.

**"Rubric Is All You Need"** (Pathak et al., ICER 2025, arXiv:2503.23989) is the most directly applicable paper: **question-specific rubrics outperform generic ones** on Spearman ρ and Cohen's κ when grading code. Translated for Jac: a per-task rubric that encodes "for *this* traversal task, the solution should use a `walker` with `visit`" beats a generic "code is idiomatic" rubric.

### Code benchmarks: copy MultiPL-E, not HumanEval
HumanEval's original pattern — function signature + docstring + hidden unit tests, scored with pass@k using the unbiased estimator `pass@k = 1 - C(n-c,k)/C(n,k)` (Chen et al. 2021, arXiv:2107.03374) — is the reference, but **MultiPL-E** (Cassano et al., IEEE TSE 2023, arXiv:2208.08227, github.com/nuprl/MultiPL-E) is the actually-relevant template because it solved the exact problem you face: translating HumanEval/MBPP to 18 languages including low-resource ones (Racket, OCaml, Lua, R, Julia). Their approach: most HumanEval tests have the form `f(v_in) = v_out` with first-order values, so tests translate mechanically. **Per-language you implement a `humaneval_to_L.py` Translator class (prompt translation + type translation + value translation) and an `eval_L.py` `eval_script` that runs code in a subprocess with a timeout.** This is ~100–300 lines per language. For Jac you would implement `humaneval_to_jac.py` and an `eval_jac.py` that shells out to `jac run` or (since Jac compiles to Python bytecode) runs the compiled artifact.

The low-resource performance picture from MultiPL-E is sobering and directly informs expected effect sizes: StarCoder/Codex performance drops sharply on Racket and other niche languages compared to Python, and **MultiPL-T** (Cassano et al., OOPSLA 2024, arXiv:2308.09895) showed fine-tuning on machine-translated corpora substantially closes the gap. Most relevantly, MultiPL-T includes **a hand-graded Racket style rubric worth 15 points across categories**, graded on 30–35 problems — this is the closest published precedent to what you are building for Jac idiomaticity, and it specifically found that "fine-tuning has a small negative impact on coding style" even as it improves correctness, which *is exactly the failure mode your eval needs to detect*: code that passes tests but isn't idiomatic. Fork this rubric structure.

**LiveCodeBench** (Jain et al., arXiv:2403.07974) is the contamination-resistance template — annotate every task with a release/creation date, allow filtering to post-cutoff items. For Jac, contamination is lower risk (little public code) but timestamp tasks anyway for future-proofing.

pass@k specifics: sample **n=200 at temperature 0.2** for pass@1 (Chen 2021 convention), or n=20 and K=10 if compute is tight. Newer work (medium.com/@ggfincke) shows performance saturates by k=5–10 on HumanEval-easy, so **pass@1 and pass@5 are the useful reporting metrics; pass@100 is largely wasted effort**.

### Idiomaticity measurement: hybrid AST rules + LLM judge
There is no metric that directly measures "idiomaticity" in the sense you want. **CodeBLEU** (Ren et al. 2020, arXiv:2009.10297) is a weighted combination of four components — standard BLEU, weighted n-gram (keywords up-weighted ~5x), AST subtree match, and data-flow match — and it *does* correlate better with human judgment than BLEU alone, but it requires a parser for the target language (tree-sitter or equivalent) and a reference answer, and its "idiomaticity" signal is really "structural similarity to reference." Given Jac has a tree-sitter grammar in the Jaseci repo, CodeBLEU-style metrics are implementable but not worth building in v0 unless you have many curated reference solutions.

**Allamanis & Sutton's HAGGIS** ("Mining Idioms from Source Code," FSE 2014, arXiv:1404.0417) and follow-up "Mining Semantic Loop Idioms" (TSE 2018) formalize code idioms as probabilistic tree-substitution grammars — syntactic fragments with metavariables that recur across projects. The more practical takeaway is their operational definition: **an idiom is a specific AST fragment you can detect with a parser**. For Jac, this maps to concrete tree-sitter queries: "does the AST contain a `walker` archetype?" "does it contain a `visit` statement?" "does it contain `++>` edge creation?" "does it use typed `edge` archetypes or dict-of-lists?" These are **binary, cheap, deterministic, and not subject to LLM-judge noise**. Compile a set of 8–12 such detectors and combine with a Prometheus-style LLM judge for the subjective part. Work on "Refactoring to Pythonic Idioms" (arXiv:2406.03660) and the "Don't Judge Code by Its Cover" paper (arXiv:2505.16222, 2025) establish that superficial-variation bias is real — so normalize formatting before judging.

The SQL analogy is instructive: Spider and BIRD grade primarily by **execution accuracy**, not query quality. BIRD (Li et al., arXiv:2305.03111) added a **Valid Efficiency Score (VES)** as a secondary metric — runtime relative to gold — but nobody has a consensus rubric for "idiomatic SQL." **This is the current state of the art, and it argues for your approach: execution-based correctness plus a separate explicit rubric for idiomaticity is the most defensible setup available.**

### Paired ablation design: this is where the biggest wins live
**Miller (2024), "Adding Error Bars to Evals,"** arXiv:2411.00640 (Anthropic) is the single most important reference here. His key formula: paired-difference variance equals unpaired variance minus `2·Cov(x_A, x_B)/n`. Because the same task under "with skill" and "without skill" will be positively correlated (~0.4–0.7), paired analysis delivers a free 30–60% standard-error reduction. **Always run the same tasks in both arms, freeze task order, freeze the rest of the system prompt, vary only the skill block.**

For a binary pass/fail intervention with paired inference at α=0.05, power 0.80, and correlation 0.5, Miller's formula gives approximate task requirements of **~25 tasks to detect a 20pp effect, ~80–100 tasks for 10pp, ~300–400 for 5pp, and ~1500+ for 2pp**. If your SKILL.md is doing real work on a low-resource language, you should expect ≥10pp, so **40–60 tasks is defensible for v0**. Back this up with **tinyBenchmarks** evidence (Polo et al., ICML 2024, arXiv:2402.14992): 100 IRT-selected items reproduce full-benchmark rankings on MMLU's 14K items, confirming that well-chosen small evals are powerful for ranking.

Use **McNemar's test for paired binary outcomes and paired bootstrap (10,000 resamples) for ordinal/continuous scores**. For small n (<100) avoid CLT-based intervals — Bowyer et al. (ICML 2025, arXiv:2503.01747) show they badly under-cover; use Bayesian Beta-Binomial credible intervals or Wilson score intervals instead. Dror et al.'s "Hitchhiker's Guide to Testing Statistical Significance in NLP" (ACL 2018) remains the protocol reference.

**Variance reduction that actually matters, in rank order**: (1) paired design with paired SE, (2) same task order / same scaffolding, (3) K≥5 samples per task averaged, (4) clustered SE if tasks come in families, (5) do *not* set temperature=0 to "reduce variance" — Miller §3.3 explicitly argues this biases results. Use the deployment temperature (0.2–0.7) and kill noise by averaging K samples.

### Skill docs: Anthropic's Agent Skills is the right specification
Anthropic published Agent Skills on Oct 16, 2025 (anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills, github.com/anthropics/skills) and made it an open standard at agentskills.io in December 2025. **The canonical structure is YAML frontmatter (`name`, `description`) + Markdown body + optional linked files loaded on demand via progressive disclosure.** Anthropic's own guidance explicitly says: *"Start with evaluation. Identify specific gaps in your agents' capabilities by running them on representative tasks and observing where they struggle or require additional context. Then build skills incrementally to address these shortcomings."* You are doing exactly this.

Adjacent precedents worth noting: **Jaseci Labs already publishes "LLMDocs" (Mini and Full variants)** at docs.jaseci.org — condensed Jac language reference materials optimized for LLM context windows. This is their internal answer to the SKILL.md problem. Your eval should probably test **SKILL.md vs. baseline vs. LLMDocs-Mini vs. LLMDocs-Full** as a 4-arm comparison, which would directly inform whether a Skills-format doc beats their existing dump. Also relevant: **Grammar Prompting** (Wang et al., NeurIPS 2023, arXiv:2305.19234) showed that including BNF grammars in context helps LLMs generate code in unfamiliar formal languages, and **Gorilla** (Patil et al., NeurIPS 2024, arXiv:2305.15334) showed retrieval-aware training for API-heavy code; both confirm that context design significantly affects low-resource code gen.

### There is no published Jac codegen benchmark
This is a genuine gap. Jason Mars' group has published multiple papers *about* Jac (MTP at OOPSLA 2025, arXiv:2405.08965; "Object-Spatial Programming," arXiv:2503.15812; SemTexts, arXiv:2511.19427; the Jaseci CAL paper 2023) — but **every single one measures task performance of LLMs *called through* Jac (`by llm()` / MTP), not LLM performance at *writing* Jac code**. Their internal eval infrastructure (SLAM at github.com/jaseci-labs/slam; SLMEval at arXiv:2505.16003) is general-purpose and could be adapted, but there's no JacHumanEval to fork. Good source repos for idiomatic reference code: `jaseci-labs/jaseci`, `jaseci-labs/jaclang` test suite, `jaseci-labs/Agentic-AI`, `jaseci-labs/jasketch`, and the Jac Book tutorials at jac-lang.org.

## Concrete v0 recipe

### Task set (build this first, ~3–4 hours)
Build **40–60 paired tasks**, stratified across four buckets of roughly equal size: (1) pure syntax/type tasks that exercise Jac's superset-of-Python features without graph primitives, (2) graph-construction tasks requiring `node`/`edge`/`++>`, (3) traversal tasks requiring `walker`/`visit`/`disengage`, (4) `by llm()` integration tasks. Each task has a prompt in HumanEval style (function signature + docstring) and hidden unit tests that execute via `jac run`. **Calibrate baseline (no-skill) pass rate to the 30–70% range** by piloting 5–10 candidates first — tasks where baseline is 0% or 100% cannot discriminate your intervention. Time-stamp every task (LiveCodeBench pattern). Hold out ~20% of tasks as a sealed test set you don't look at while iterating.

### Test harness (use Inspect AI)
**Inspect AI** (github.com/UKGovernmentBEIS/inspect_ai) is the right harness — MIT-licensed, production-grade, Dataset→Solver→Scorer abstraction with Docker sandboxes built in, and it's what Anthropic and DeepMind use. One `Task` per Jac scenario; a custom `Solver` generates completions with and without the skill; a sandboxed `Scorer` runs `jac run` against hidden tests for correctness and calls a separate model-graded scorer for idiomaticity. Log everything for audit. If Inspect AI is too heavy for the next few hours, the minimum viable version is: Python subprocess running `jac run` with a 30-second timeout inside a Docker container, plus a separate Python script that calls the LLM judge on each solution.

### Correctness scoring (40% weight)
Unit-test pass/fail per task per sample. Use `pass@1` as primary (n=5–10 samples at temp=0.2). Report `pass@5` as secondary. Unbiased estimator is mandatory: `pass@k = 1 - C(n-c,k)/C(n,k)`. **Do not use naive `1-(1-p̂)^k`** — it's biased and the research community flags this.

### Idiomaticity scoring (60% weight) — the hard part
**Two-stage hybrid, mirroring the MultiPL-T Racket rubric precedent**:

**Stage 1 — deterministic AST checks via tree-sitter-jac** (free, fast, reliable). Build 8–12 binary detectors returning yes/partial/no, weighted into a 0–1 subscore. Examples to include: uses at least one `node` archetype; uses at least one typed `edge` archetype (not dict-of-list); uses a `walker` for traversal (not manual recursion/BFS on dicts); uses `visit` statement; uses `disengage` where halting logic exists; uses `++>`/`<++>` for edge creation (not direct field assignment); type annotations present on all `has`, params, returns; uses `by llm()` where LLM integration is requested (not hardcoded prompt strings); uses filter/assign comprehensions where appropriate; avoids Python-style re-implementation of graph primitives. These are the *specific Jac-native constructs* from the language reference and the OSP paper (arXiv:2503.15812).

**Stage 2 — LLM judge** with Jac language reference (Jaseci's LLMDocs-Mini) in system prompt. Use Prometheus format: task description, gold reference solution when available, candidate code, explicit rubric with descriptors per level 1–5 (score 1 = "near-direct Python port," score 5 = "idiomatically uses nodes/edges/walkers/visit/abilities, leverages object-spatial programming"). Require JSON output with per-criterion scores and `[RESULT] X` final score. **Structure the prompt so the judge first lists which Jac constructs appear** (priming against Python-bias), then evaluates each rubric item citing line numbers, then assigns the overall score.

**Judge configuration**: pointwise reference-based when gold answer exists (most reliable per Zheng 2023); pointwise reference-free otherwise. Use a different model family than the generator (if generating with Claude, judge with GPT-4 or Gemini-2.5-Pro). For high-stakes tasks, ensemble 2 judges and average. **Explicitly instruct the judge to penalize Python-transliterated patterns** and reward Jac-native constructs even when more verbose — this counteracts the perplexity-familiarity bias that will otherwise push scores toward Python-like code.

**Combine**: per-task idiomaticity = 0.4 × AST-subscore + 0.6 × LLM-judge-score (normalized to 0–1). Overall task score = 0.4 × correctness + 0.6 × idiomaticity, per your weighting.

### Judge validation (non-negotiable; ~1–2 hours)
Hand-grade 30–50 Jac snippets spanning the quality spectrum. Measure the judge's Cohen's κ and Spearman ρ against your hand labels. **Target κ ≥ 0.4** (substantial agreement by Landis-Koch). If κ < 0.4, iterate the rubric — shorten CoT if false negatives dominate, sharpen descriptors if descriptor ambiguity dominates. Re-validate after every rubric revision. Eugene Yan: *"I often see human inter-rater reliability being as low as 0.2–0.3,"* so κ≥0.4 is a competitive judge.

### Statistical reporting (defensible delta claims)
Per-task: mean correctness (of K samples) and mean idiomaticity per arm. Aggregate: paired mean Δ per dimension with paired bootstrap 95% CI (10,000 resamples). Also report **McNemar's test** on pass/fail outcomes and **Wilcoxon signed-rank** on ordinal scores. For n < 100, add Bayesian Beta-Binomial credible intervals (Bowyer 2025). Report the 2×2 flip table (pass→pass / pass→fail / fail→pass / fail→fail) — this is more informative than any summary number. Report baseline pass rate to contextualize ceiling/floor. Report **correlation between arms** (high correlation = paired analysis gave you power; low = something weird happened).

### Controls you must run
(1) **Irrelevant-context control**: re-run with a SKILL.md for a *different* language to prove the improvement isn't just "more tokens." (2) **No-op noise floor**: run the baseline twice with different seeds to bound variance from stochasticity alone — any claimed effect must exceed this floor. (3) **Pre-commit to the effect size that would matter** (e.g., "we ship if Δ_overall ≥ +8pp") *before* looking at results to avoid p-hacking.

## Pitfalls to avoid

**Don't use a fine-tuned open judge (Prometheus, PandaLM, JudgeLM, Auto-J) as a drop-in for Jac.** Huang et al. (ACL Findings 2025, arXiv:2403.02839) show these collapse outside their trained schemes. Use their *prompt format* with a strong proprietary model instead. **Don't use CodeBLEU as your primary idiomaticity metric** — it measures structural similarity to a reference, not idiom usage, and it's noisy without many references. **Don't set temperature=0 for determinism** (Miller §3.3). **Don't report unpaired confidence intervals** after running paired — you throw away the correlation and your error bars balloon 2x. **Don't rely on a single LLM judge run per task** — at minimum run 3 judgments and take the median; self-consistency stabilizes scores. **Don't forget to strip/normalize comments and formatting** before judging (arXiv:2505.16222) — superficial-variation bias is documented and will inject noise. **Don't claim the effect generalizes beyond your task distribution** — report per-stratum deltas (syntax / graph-construction / traversal / LLM-integration) so readers can see where SKILL.md helps vs. doesn't. **Don't skip validating the judge against hand labels** — an unvalidated judge can have κ below 0.1 on code (arXiv:2507.16587).

## Where consensus is strong vs. unsettled

**Strong consensus, act on it**: (a) execution-based correctness beats any LLM judge for correctness; (b) position bias is real and swap-and-tie is the standard fix; (c) binary/short-Likert rubrics beat 1–10 scales; (d) paired design with paired SE gives large free variance reduction; (e) Chen et al.'s unbiased pass@k formula is standard; (f) MultiPL-E's per-language translator pattern is the only scalable recipe for adding a new language; (g) progressive disclosure (Anthropic Skills spec) is the right format for skill docs.

**Unsettled, hedge with multiple methods**: (a) whether LLM judges can be trusted for code at all without execution — Cohen's κ ranges from 0.1 to 0.7 across studies, so validate against your own hand labels; (b) pairwise vs. pointwise for code specifically — CodeJudge uses pointwise successfully, but some SE studies prefer pairwise; run both if time permits; (c) minimum viable N — tinyBenchmarks argues ~100 IRT-selected items suffice, Miller argues ~1,000 for general benchmarks; for a targeted ablation expecting a large effect, 40–60 is defensible but fragile; (d) whether to normalize formatting before judging — "Don't Judge Code by Its Cover" says yes, but this may erase legitimate style signal.

## Conclusion: what makes this eval defensible

A v0 eval on a niche language will not have the statistical power of HumanEval's 164×many-runs or SWE-bench's 2,294 tasks. What makes it credible is **architectural rigor**: paired design, execution-based correctness, a hybrid AST+LLM idiomaticity score that can't be gamed by pure syntax mimicry, validated judge agreement against hand labels, bootstrap CIs appropriate for small n, a pre-committed effect-size threshold, and irrelevant-context controls that rule out the "more tokens" confound. The single highest-leverage investment is **the 30–50 hand-labeled Jac examples you grade yourself to validate the judge** — without this, any claim about SKILL.md's effect rests on a judge that may be no better than chance. With it, you have a defensible result regardless of which direction the delta points, and you will have built the first published-quality Jac codegen eval, which is itself a genuine research contribution since none exists today.
